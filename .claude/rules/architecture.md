# Architecture rules вЂ” DirectorioLocal CR

This is the source of truth for architectural decisions. All other agent and rule files defer to this when there's any conflict. Verified against current Astro 6 and Supabase docs (2026).

---

## 1. Rendering strategy

### Default: pure SSG
`output: 'static'` (Astro default). Every directory page (canton, district, category listing, provider profile) is pre-rendered at build time. This is the SEO-critical path вЂ” Google must see the full HTML on first request.

### Server Islands for dynamic chunks
Use `server:defer` on components whose content changes faster than the build cycle:
- Reviews list on a provider profile (changes whenever a new review is posted)
- "Recently active providers in this district" widgets
- Aggregated rating numbers if they update frequently

The page shell stays static (good for SEO); the island fetches on demand at request time.

### Migration path
- **Phase 1 (MVP)**: pure SSG, no adapter, deploy as static site to Vercel
- **Phase 2 (Astro Actions for forms)**: add `@astrojs/vercel` adapter, switch to `output: 'server'`, mark every page `export const prerender = true` except the few that need server rendering
- **Phase 3 (build > 5 min)**: enable Vercel ISR with `adapter: vercel({ isr: true })` so changes propagate without full rebuild
- **Phase 4 (auth, geo redirects)**: add Astro middleware with `middlewareMode: 'edge'`

`output: 'hybrid'` no longer exists in Astro 6. Don't reach for it.

## 3. TypeScript & type-safe Supabase

### Generate types from the live schema
```bash
npx supabase gen types typescript --project-id <ref> > src/types/database.types.ts
```
Run this whenever the schema changes. Wire it as a script in `package.json`:
```json
{ "scripts": { "db:types": "supabase gen types typescript --project-id <ref> > src/types/database.types.ts" } }
```

### Typed singleton client
```ts
// src/lib/supabase.ts
import { createClient } from '@supabase/supabase-js'
import type { Database } from '@types/database.types'

export const supabase = createClient<Database>(
  import.meta.env.PUBLIC_SUPABASE_URL,
  import.meta.env.PUBLIC_SUPABASE_ANON_KEY
)
```

### Strict TypeScript
- `tsconfig.json` extends `astro/tsconfigs/strict`
- No `any` outside generated files
- All env vars typed in `src/env.d.ts`

## 4. Data access pattern

### Hard rules
- **Every query lives in `src/lib/queries/<domain>.ts`** вЂ” never inline in `.astro` frontmatter
- **No `select('*')` in production** вЂ” list columns explicitly. Better caching, smaller payloads, type narrowing.
- **Always destructure both `data` and `error`** from Supabase responses
- **Errors fail loud, not silent** вЂ” return `null` or `[]` only when "not found" is a legitimate state, never to swallow real errors. Log unexpected errors before returning the fallback.

### Function shape
```ts
// src/lib/queries/providers.ts
export async function getProvidersByDistrictAndCategory(
  districtSlug: string,
  categorySlug: string
): Promise<Provider[]> {
  const district = await resolveDistrict(districtSlug)
  if (!district) return []

  const category = await resolveCategory(categorySlug)
  if (!category) return []

  const { data, error } = await supabase
    .from('providers')
    .select('id, name, phone, whatsapp, description, photo_url, provider_categories!inner(category_id)')
    .eq('district_id', district.id)
    .eq('provider_categories.category_id', category.id)
    .eq('verified', true)
    .order('created_at', { ascending: false })

  if (error) {
    logger.error('getProvidersByDistrictAndCategory', { districtSlug, categorySlug, error })
    return []
  }
  return data ?? []
}
```

## 5. Database & RLS

### Schema additions vs the original CLAUDE.md
Add `owner_id uuid REFERENCES auth.users(id)` to `providers`. This is what RLS enforces against вЂ” without it, "users can edit their own provider" is impossible.

### Required indexes
```sql
CREATE INDEX idx_providers_district ON providers(district_id);
CREATE INDEX idx_providers_verified ON providers(verified) WHERE verified = true;
CREATE INDEX idx_provider_categories_category ON provider_categories(category_id);
CREATE INDEX idx_districts_canton ON districts(canton_id);
CREATE INDEX idx_districts_slug ON districts(slug);
CREATE INDEX idx_cantons_slug ON cantons(slug);
CREATE INDEX idx_categories_slug ON categories(slug);
```

### RLS policies (Postgres-first)
Every table has RLS enabled, no exceptions. Templates:
```sql
-- Public read for verified content
ALTER TABLE providers ENABLE ROW LEVEL SECURITY;

CREATE POLICY "providers_public_read" ON providers
  FOR SELECT USING (verified = true);

-- Authenticated users see their own unverified row
CREATE POLICY "providers_owner_read_unverified" ON providers
  FOR SELECT USING (auth.uid() = owner_id);

-- Insert only with own owner_id
CREATE POLICY "providers_self_insert" ON providers
  FOR INSERT WITH CHECK (auth.uid() = owner_id);

-- Update only own rows, never flip verified themselves
CREATE POLICY "providers_owner_update" ON providers
  FOR UPDATE USING (auth.uid() = owner_id)
  WITH CHECK (auth.uid() = owner_id AND verified = (SELECT verified FROM providers WHERE id = providers.id));
```

### Service role
Only used via Edge Functions or admin scripts. **Never** in frontend code. **Never** in `PUBLIC_*` env vars.

## 6. SEO architecture

### URL hierarchy
| Path | Page | JSON-LD |
|---|---|---|
| `/` | Home with search | `WebSite` + `SearchAction` |
| `/{canton}/` | Canton landing | `Place` + `BreadcrumbList` |
| `/{canton}/{distrito}/` | District landing | `Place` + `BreadcrumbList` |
| `/{canton}/{distrito}/{categoria}` | Provider listing | `ItemList` + `BreadcrumbList` |
| `/proveedor/{id}-{name-slug}` | Provider profile | `LocalBusiness` |

The `id-name-slug` form on provider URLs gives both stable ID lookups and human-readable URLs. Slug component is decorative (don't validate it; `id` resolves the row).

### Per-page SEO requirements
Every page must have:
- Unique `<title>` (50-60 chars), keyword-loaded with location
- `<meta name="description">` (150-160 chars) including a local landmark or reference
- `<link rel="canonical">`
- Single `<h1>` with the primary keyword
- JSON-LD per the table above
- Open Graph tags (og:title, og:description, og:image, og:type)

### `LocalBusiness` schema must include
- `name`, `telephone`, `address` (PostalAddress with `addressLocality`, `addressRegion`, `addressCountry: 'CR'`)
- `areaServed` вЂ” array of district names (this is what 2026 AI overviews use for hyperlocal matching)
- `geo` (GeoCoordinates) when known
- `openingHoursSpecification` when known
- `aggregateRating` when в‰Ґ3 reviews
- `image`, `url`, `description`

### Sitemap
Use `@astrojs/sitemap`:
```js
sitemap({
  entryLimit: 10000,
  serialize(item) {
    if (/\/proveedor\//.test(item.url)) {
      item.priority = 0.7
      item.changefreq = 'weekly'
    } else if (/\/[^/]+\/[^/]+\/[^/]+$/.test(item.url)) {
      item.priority = 0.9
      item.changefreq = 'daily'
    }
    return item
  },
})
```
A dynamic `src/pages/robots.txt.ts` API route points to `sitemap-index.xml`.

### Anti-thin-content
Programmatic SEO works in 2026 only if pages have real content. Enforce:
- **Skip listings with <3 providers** in `getStaticPaths()`. A page with one provider is not a directory page; redirect or omit.
- **Per-district category descriptions** (not generic). Even one sentence with a local landmark beats a templated paragraph.
- **Internal linking automation**: at the bottom of every listing, link to nearby districts with the same category and other categories in the same district. This builds the topical mesh search engines reward.

## 7. Forms вЂ” Astro Actions, not API routes

Astro Actions replace API routes for frontend-driven mutations. Benefits over `src/pages/api/*`:
- Type inference end-to-end (client sees the same input/output types as the handler)
- Built-in Zod validation
- Standardized `ActionError` for client error handling
- Less boilerplate (no manual `request.formData()` parsing)

Use Astro Actions for:
- Provider registration form
- Review submission
- Photo upload (file input в†’ Supabase Storage from the action handler)

Use API routes only for:
- External webhooks (Stripe, etc.)
- Dynamic `robots.txt`, `feed.xml`, etc.

## 8. Build performance

### Combinatorial explosion
Naive `getStaticPaths` for `[canton]/[distrito]/[categoria]` = `O(cantons Г— districts Г— categories)`. Mitigations in priority order:
1. **Filter to combinations with в‰Ґ1 provider** (also satisfies anti-thin-content)
2. **Parallelize fetches** with `Promise.all` inside `getStaticPaths`
3. **Cache** the combinations query in a build-time variable shared across pages
4. **Migrate to ISR** if build time exceeds ~5 minutes

### Image optimization
- `<Image>` from `astro:assets` only (forbid bare `<img>`)
- Remote images (Supabase Storage) require explicit `width` and `height`
- Allowlist Supabase host in `astro.config.mjs`:
  ```js
  image: { domains: ['<project>.supabase.co'] }
  ```
- Lazy-load below-fold images (`loading="lazy"`)

## 9. Search

### Phase 1: Postgres full-text search
A Supabase RPC function with `to_tsvector('spanish', name || ' ' || description)` and a GIN index. Called via `supabase.rpc('search_providers', { q: '...' })`.

### Phase 2: filter combinators
URL params `?cat=fontaneria&canton=cartago` drive both server-side filtering and the visible filter UI.

### Phase 3 (future): vector search
Postgres `pgvector` extension or Supabase Storage Vectors for "providers similar to this one" recommendations. Not needed for MVP.

## 12. Programming standards

### Tooling
- **TypeScript strict** (`extends: 'astro/tsconfigs/strict'`)
- **ESLint** with `eslint-plugin-astro` and `@typescript-eslint`
- **Prettier** with `prettier-plugin-astro`
- **Husky + lint-staged** for pre-commit checks
- **`@astrojs/check`** for type-checking in CI

### Naming
- Files: kebab-case for pages, PascalCase for components, camelCase for utilities
- DB columns: snake_case (Postgres convention)
- TS variables/functions: camelCase
- Types/interfaces: PascalCase
- Constants: SCREAMING_SNAKE_CASE

### Banned patterns
- `console.log` in production code (use `logger.ts` with dev-only output)
- Hardcoded canton/district/category IDs (always derive from URL slug)
- Direct `fetch()` to Supabase URLs (always go through the typed client)
- Business logic in `.astro` frontmatter (push it down to `src/lib/queries/`)
- `select('*')` (list columns explicitly)
- Mocked Supabase in tests (use real DB)
