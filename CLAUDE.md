# DirectorioLocal CR

Hyperlocal "yellow pages"-style services directory for Costa Rica. It launches with a single district, but the architecture supports every canton/district in the country from day one.

## Stack
- **Frontend**: Astro 5 + Alpine.js (no heavy JS frameworks, SSG-first, SEO-first)
- **Database**: Supabase (PostgreSQL + Auth + Storage + auto-generated API)
- **Hosting**: Vercel (deploy via git push)
- **Styling**: Tailwind CSS
- **Language**: TypeScript strict

## Key architecture
- URLs: `/{canton}/{district}/{category}` — example: `/cartago/san-nicolas/fontaneria`
- Multi-tenancy by data, NOT by code: a single deploy serves all of CR, filtered by `canton_id` / `district_id`
- Each provider has: `district_id`, categories (many-to-many), `owner_id` (auth.users), photo, phone, WhatsApp
- No custom backend: the frontend uses the typed Supabase JS client directly; mutations go through Astro Actions
- SEO-first: Astro generates static HTML, Google indexes by canton/district/category. `LocalBusiness` schema with `areaServed` for hyperlocal AI overviews (2026 best practice).

## Project structure
```
src/
├── pages/                          # Routes — kebab-case, [param].astro for dynamic
│   ├── index.astro
│   ├── [canton]/
│   │   ├── index.astro             # Canton landing
│   │   ├── [distrito]/
│   │   │   ├── index.astro         # District landing
│   │   │   └── [categoria].astro   # Filtered listing
│   ├── proveedor/[id].astro
│   └── api/                        # External webhooks only
├── components/                     # PascalCase.astro — presentational only
│   ├── ui/                         # Button, Card, Badge, Input
│   ├── directory/                  # ProviderCard, CategoryGrid, SearchBar
│   └── seo/                        # JsonLd, OpenGraph, Breadcrumbs
├── layouts/                        # BaseLayout, DirectoryLayout, ProviderLayout
├── lib/                            # All business logic + DB access
│   ├── supabase.ts                 # Singleton typed client
│   ├── queries/                    # One file per domain (providers, categories, geography)
│   ├── seo/                        # JSON-LD generators
│   ├── slug.ts
│   └── logger.ts
├── actions/                        # Astro Actions — server-side mutations with Zod
│   └── index.ts
├── types/
│   └── database.types.ts           # Generated by `supabase gen types` — never hand-edit
└── env.d.ts
```

Path aliases (`@components/*`, `@layouts/*`, `@lib/*`, `@types/*`, `@actions/*`) are configured in `tsconfig.json`.

## Data model (Supabase)
- `cantons` (id, name, slug)
- `districts` (id, canton_id, name, slug)
- `categories` (id, name, slug, icon_emoji)
- `providers` (id, name, phone, whatsapp, email, description, photo_url, district_id, **owner_id** uuid REFERENCES auth.users, verified, created_at)
- `provider_categories` (provider_id, category_id) — N:N relation
- `reviews` (id, provider_id, author_id uuid REFERENCES auth.users, rating, comment, created_at)

All tables have RLS enabled. `providers` and `categories` are publicly readable when `verified = true`; writes are gated to `auth.uid() = owner_id`.

## Conventions
- Slugs are always kebab-case and unaccented (use `toSlug` from `@lib/slug`)
- Every Supabase query lives in `src/lib/queries/<domain>.ts` — never inline in `.astro` frontmatter
- Every query on `providers` filters by `district_id` minimum
- No `select('*')` in production — list columns explicitly
- Provider images go in Supabase Storage, public bucket `provider-photos`
- Environment variables: `PUBLIC_SUPABASE_URL` and `PUBLIC_SUPABASE_ANON_KEY` in `.env`. `SUPABASE_SERVICE_ROLE_KEY` in Vercel only — never `PUBLIC_*`, never committed.
- Mutations (registration, reviews, photo upload) go through Astro Actions in `src/actions/`, not API routes
